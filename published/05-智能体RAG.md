在上一章中，我们赋予了 AI 智能体调用外部工具的能力，让它拥有了“三头六臂”。但在许多企业级应用中，最宝贵的资产并非外部的公共 API，而是公司内部堆积如山的文档、政策手册和历史知识库。

如何让 AI 准确地读取这些内部资料来回答问题？这就引出了当今 AI 领域最火爆的技术之一：**RAG（检索增强生成，Retrieval-Augmented Generation）**。而本章，我们将不满足于基础的 RAG，我们要探索它的终极进化形态——**智能体 RAG (Agentic RAG)**。

<a id="ch05-1"></a>
## 5.1 从基础 RAG 到智能体 RAG 的进化

为了理解智能体 RAG 的强大，我们先来复习一下什么是基础 RAG。
想象一下，大语言模型 (LLM) 就像一个肚子里装满公共知识的学霸，但如果你问它你们公司昨天刚发布的《员工差旅报销规定》，它必然一头雾水，因为它没有这部分训练数据。基础 RAG 的做法，就是给这个学霸发一本“参考书”：当用户提问时，系统先去数据库里搜索相关的文档段落，然后把这些段落连同问题一起塞给大模型，让它“开卷考试”作答。

**但是，基础 RAG 有一个致命弱点：它太死板了。** 它只是简单地执行“搜索 -> 回答”的线性流程。如果搜出来的文档残缺不全，大模型可能依然会胡编乱造；如果用户的问题极其复杂，简单的搜索根本无法覆盖全貌。

这就是 **智能体 RAG (Agentic RAG)** 登场的时候了。它为传统的 RAG 注入了智能体的灵魂：
1.  **任务拆解与规划**：当接收到一个复杂查询时，智能体 RAG 不会傻乎乎地直接去搜。它会先分析查询，将其拆解为一系列较小的子任务，并规划需要调用哪些数据源和工具[1, 2]。
2.  **自我验证 (Self-Verification)**：这是它最强大的超能力。当它从数据库中检索到信息后，它会主动“反思”：这些信息足够回答用户的问题吗？如果不够，它绝不敷衍了事，而是会触发“工具调用”，去寻找额外的数据，直到拼凑出完整的答案。
3.  **记忆与自适应**：智能体能够保持长期的记忆，记住之前尝试检索的过程并在下次改进。正如课程主讲人打趣说的那样：“我本想管这个叫‘自适应 RAG (Adaptive RAG)’的，可惜给这些技术起名字的时候没人来问我的意见，所以现在大家只能叫它智能体 RAG 了！”。

## 5.2 实战演练：当文档检索遇上工具调用

理论讲究“融会贯通”，让我们回到代码编辑器，看看在使用 Semantic Kernel 和 Azure AI Search 的项目中，智能体 RAG 是如何大显身手的。

### 精心设计的“提示词插件”
在代码中，我们引入了一个特殊的 `prompt plugin`（提示词插件）。这个插件在系统检索到文档后，会动态地增强大模型的提示词。
我们给大模型下达了明确的指令：“**首先审查检索到的上下文 (Retrieve Context) 以查看它是否回答了用户的查询；如果找不到答案，请尝试使用任何可用的插件函数！**”。这就是在代码层面激活“自我验证”机制的秘诀。

### 应对复杂查询：检索与工具的完美双打
我们往 Azure AI Search 数据库里塞入了一些虚拟公司 (Contoso) 的旅游保险政策和目的地介绍文档。同时，为了弥补静态文档的不足，我们还给智能体配备了一个查询实时天气的工具：`weather_info_plugin`。

现在，让我们看看智能体如何应对不同难度的挑战：

**回合一：纯文档检索**
用户问：“请解释一下 Contoso 的旅游保险覆盖范围。”
智能体启动，检索到了最高相关度的文档（保险政策文档），发现信息足以回答问题，于是完美地生成了回复。这就像是一次满分的基础 RAG 演示。

**回合二：文档缺失，工具补位**
用户问：“马尔代夫的平均气温是多少？”
智能体去搜索文档，虽然找到了“马尔代夫”是公司提供的目的地，但文档里根本没写温度（毕竟气候数据是动态的）。
此时，**自我验证**触发！智能体发现文档不足以回答查询，立刻扭头去调用了 `get_destinations`（或关联的温度查询函数）。拿到温度数据后，它将检索到的文档背景与 API 返回的温度融合，给出了精确的答案。

**回合三：终极考验——多步复合推理**
用户抛出了一个极其刁钻的问题：“**Contoso 提供哪些好冷的旅游胜地？那里的平均气温是多少？**”
这是一个典型的需要多步操作的复杂查询。请看智能体令人惊叹的表演：
1.  **第一步（检索）**：它首先去数据库检索，查到公司提供的热门目的地包括“马尔代夫、瑞士阿尔卑斯山 (Swiss Alps) 和非洲 Safari”[7]。
2.  **第二步（逻辑判断与工具调用）**：它利用自身常识判断出“瑞士阿尔卑斯山”是寒冷的目的地。紧接着，为了获取温度，它自动调用 API 工具，传入参数“Swiss Alps”查询温度[7]。
3.  **第三步（完美融合）**：它将两路信息汇合，输出最终回答：“Contoso 提供的一个极佳的寒冷目的地是瑞士阿尔卑斯山，其平均气温为...”[7]。

## 5.3 本章小结

在这一章，我们见证了 RAG 技术的华丽转身。智能体 RAG 彻底打破了“数据库里有什么就只能回答什么”的死板设定。通过赋予它拆解任务的智慧、自我验证的严谨以及随时调用外部工具的灵活性，我们真正打造出了一个能够自主查阅“参考书”并举一反三的超级助理。

但是，随着智能体的能力越来越强，它接触到的企业内部数据和调用的外部 API 也越来越多。这就引出了一个极其严肃的问题：**如果它失控了怎么办？如果它泄露了用户隐私怎么办？**
在下一章“构建值得信赖的智能体 (Building Trustworthy AI Agents)”中，我们将化身安全专家，探讨如何给这位神通广大的数字员工，戴上安全与道德的紧箍咒！

> **引用参考**

> *   **基础 RAG (Basic RAG)**: 检索增强生成。允许 LLM 将用户的查询发送到数据库，检索相关信息并添加到上下文中，以提供基于外部知识的回答。
> *   **Agentic RAG (智能体 RAG)**: 结合了 RAG 与智能体特性的架构。它包含任务分析、自我验证以及在检索信息不足时动态调用工具的能力[1, 2]。
> *   **Azure AI Search**: 微软提供的企业级云端搜索服务。在本章代码实例中被用作存储和检索非结构化旅游文档的向量/混合数据库。
