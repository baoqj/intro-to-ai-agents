在上一章中，我们为 AI 智能体戴上了“安全帽”，确保它在执行任务时值得信赖且不会越界。现在，你的数字员工已经准备好接手真正的大项目了。

想象一下，你给智能体下达了一个极其宏大的任务：“给我安排一家四口去温暖的地方玩三天”。
如果是一个初级的聊天机器人，它可能会立刻开始瞎编一段毫无逻辑的文字；但一个成熟的 AI 智能体，第一反应应该是像一位专业的项目经理一样，深吸一口气，然后打开 Excel 建立 WBS（工作分解结构）。

这种“谋定而后动”的智慧，在 AI 开发领域被称为**规划设计模式 (Planning Design Pattern)**[1]。

<a id="ch07-1"></a>
## 7.1 什么是规划设计模式？

正如视频课程中所指出的，智能体的一个核心特性就是能够接收任务，并制定完成该任务的计划[1]。

规划设计模式将这一特性具象化了：它强制要求 AI 智能体明确列出构成一个复杂任务的所有**子任务 (Subtasks)**[1]。
例如，去温暖的地方度过三天假期，这个听起来简单的愿望，实际上需要被无情地拆解为多个具体的执行步骤：预订机票、预订酒店、安排当地交通以及策划个人活动[1, 2]。

**为什么拆解如此重要？**

因为在企业级应用中，真正干活的往往不是一个“全能神”，而是一群各司其职的专家。当我们把大任务拆解为子任务后，就可以将不同的子任务分配给专门的下游智能体（比如专门负责买机票的 Agent，或专门负责订酒店的 Agent），甚至直接交给传统的后端代码去执行[2]。

<a id="ch07-2"></a>
## 7.2 协同作业的基础：结构化与校验

为了让下游的其他智能体或传统软件系统能够读懂大模型生成的计划，我们绝对不能让大模型输出一段随性的散文。我们需要的是极其严谨、可被代码解析的结构化数据（通常是 JSON 格式）[2]。

为了实现这一点，在 Python 开发环境中，我们通常会引入著名的 `Pydantic` 库[3]。`Pydantic` 能够为大模型的输出应用严格的规则和数据验证，确保我们拿到了所有必须的信息，才能让智能体系统继续流转[2, 3]。

<a id="ch07-3"></a>
## 7.3 代码实战：让大模型成为项目经理

让我们穿上程序员的格子衫，打开本章的代码示例（位于 GitHub 仓库的 Chapter 7 目录），看看如何用代码逼迫大模型交出一份完美的计划书。

### 1. 定义数据结构

首先，我们使用 `Pydantic` 定义出我们期望的数据模型。我们需要告诉大模型：“你必须按照这个模板来填空”。

```python
from pydantic import BaseModel
from typing import List

# 定义子任务 (Subtask) 的结构
class Subtask(BaseModel):
    assigned_agent: str   # 指定由哪个智能体来处理这个任务 (例如：flight_booking, hotel_booking)
    task_details: str     # 具体的任务描述

# 定义整体旅行计划 (Travel Plan) 的结构
class TravelPlan(BaseModel):
    main_task: str        # 用户的总体旅行请求
    subtasks: List[Subtask] # 拆解后的子任务列表
```

**代码解析**：我们定义了一个两层结构。最外层是整体计划，里面包含一个子任务的列表。每个子任务都必须明确指派给特定的智能体，并附带详细说明。

### 2. 注入“公司组织架构”

在给智能体的系统提示词中，我们必须告诉它手下有哪些“兵”可用。我们向大模型罗列了可用的智能体名单及其简介：航班预订 (flight booking)、酒店预订 (hotel)、租车 (car rental)、活动策划 (activities)、目的地信息查询 (destination info)，以及一个用于处理常规请求的默认智能体。

### 3. 见证魔法：大模型的常识推理
现在，用户输入了一个充满细节的请求：“**为带着两个孩子的家庭制定一个从新加坡去墨尔本的旅行计划 (create a travel plan for family of two kids from Singapore to Melbourne)**”。

运行代码后，得益于 `Pydantic` 的强制校验，智能体返回了一个完美的 JSON 结构。最令人惊叹的部分，藏在它的任务细节 (task details) 里。请看大模型生成的酒店预订子任务：
```json
{
  "assigned_agent": "hotel_booking",
  "task_details": "Find and book family-friendly accommodation in Melbourne for 2 adults and 2 kids."
}
```

**你发现亮点了吗？** 用户只说了“**两个孩子**”，但大模型在将任务分配给“酒店预订智能体”时，利用其强大的自然语言理解能力，**自主在任务细节中加入了“寻找家庭友好型住宿 (family-friendly accommodation)”这一极其关键的限定词**。

这就是大模型作为项目经理的恐怖实力！你可以想象这在实际业务中有多么强大：下游的系统收到这个 JSON 后，可以直接提取 `family-friendly` 作为数据库查询的过滤标签，从而精确筛选出带有儿童乐园和婴儿床的酒店。大模型不仅完成了任务拆解，还在拆解的过程中注入了极具价值的业务常识。

<a id="ch07-4"></a>
## 7.4 本章小结

通过本章的学习，我们掌握了规划设计模式。我们学会了如何使用 `Pydantic` 这样的工具来约束和验证大模型的输出，强制它将复杂任务拆解为结构化的子任务列表，并智能地分配给相应的角色。这让大模型不再只是一个空谈家，而变成了一个能写出严谨执行方案的规划师。

但是，现在图纸画好了，任务也分配下去了，那些被点名干活的“下游智能体”们（比如负责机票的、负责酒店的）该如何在一起工作呢？如果前台智能体推荐的计划，后台智能体觉得不靠谱怎么办？

这就涉及到了智能体的高阶玩法。在下一章“多智能体设计模式 (Multi-Agent Design Pattern)”中，我们将把你带入一个由多个 AI 组成的“虚拟会议室”，看看它们是如何互相协作、甚至激烈辩论的！

> **引用参考**
>
> *   **规划设计模式 (Planning Design Pattern)**: 一种让 AI 智能体在执行操作前，先将复杂任务分解为一系列清晰、可执行子任务的架构模式。
>
> *   **Pydantic**: Python 中极受欢迎的数据验证库。在本章中，它被用于强制大语言模型输出符合严格类型定义的 JSON 数据结构，从而确保下游系统能够安全地解析数据。
>
> *   **子任务 (Subtasks)**: 构成总体任务的最小执行单元，在本课示例中包含 assigned_agent (指派智能体) 和 task_details (任务详情) 两个核心字段。
