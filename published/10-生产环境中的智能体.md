## 10.1 全链路评估：不只是看看最终输出

当你评估一个传统的分类算法时，你只需要看它的准确率。但评估一个 AI 智能体，如果只盯着最终输出看，那无异于盲人摸象。

将智能体投入生产的第一步，是建立**全链路的评估体系 (Step-by-Step Evaluations)**。因为智能体的工作是一个多步流转的系统，你必须在它运行的每一个节点都设立“检查站”[2]：
1.  **初始连接与响应**：大模型服务的连接是否稳定？响应时间是多少？更换不同模型（如 GPT-4o 换成 Claude 3.5）会如何影响整体延迟[2]？
2.  **意图识别 (Intent Identification)**：智能体是否准确理解了用户的真实目的？这是确保它能完成既定任务的基石[2]。
3.  **工具选择 (Tool Selection)**：它有没有为了查天气而错误地去调用订机票的 API[2]？
4.  **工具响应质量**：外部工具返回的数据是否有错误？是否存在格式畸形 (malformed responses) 的问题[1, 2]？

**持续反馈机制**
没有反馈，就没有进化。在生产环境中，你必须在 UI 界面中为用户提供直观的反馈机制，比如最经典的“点赞/踩 (thumbs up/thumbs down)”按钮，或者满意度调查[1]。结合人工抽检和“用大模型做裁判 (LLM-as-a-judge)”的自动化评估，你就能精确测量每一次代码迭代或模型替换带来的实际影响[1]。

## 10.2 应对混乱：弹性设计与 API 错误处理

现实世界是不完美的。你依赖的外部天气接口可能会挂掉，你公司的航班数据库凭证可能会突然过期。如果你的智能体一遇到报错就直接崩溃并给用户抛出一堆红色的堆栈信息，那用户体验将是灾难性的。

生产级智能体必须具备**弹性 (Resiliency)**。让我们回到代码中，看看如何处理极其常见的 `HTTP 404` 错误[3]。

### 代码实战：配置“备用路线”

在这个模拟生产环境的例子中，用户要求预订去巴厘岛的航班，但很不巧，我们的主航班查询接口 `get_flight_times` 宕机了[3, 4]。

如果不做任何处理，对话就此终结。但作为成熟的架构师，我们在代码中为智能体配置了备用函数 (Backup Function)。

```python
# ==========================================
# 生产环境弹性设计：API 故障的降级处理
# ==========================================

# 1. 易碎的主函数 (模拟生产环境中的 API 故障)
def get_flight_times(destination: str):
    # 模拟外部服务宕机，抛出 404 错误
    raise Exception("HTTP Error 404: Flight service is currently down.")

# 2. 稳定的备用函数 (可能是较慢的数据库直连，或第三方备用服务)
def get_flight_times_backup(destination: str):
    # 提供备用数据
    return {"destination": destination, "times": ["08:30 AM", "10:45 PM"]}

# 3. 在系统提示词中定义错误处理策略
agent_instructions = """
你是一个生产级航班预订助手。
【错误处理策略】：
当你尝试调用 `get_flight_times` 遇到故障或返回 404 错误时，
请不要向用户报错！请立即自动切换调用 `get_flight_times_backup` 备用函数来获取数据。
"""

```

**运行轨迹解析**： 当用户说“帮我订一张去巴厘岛的机票”时，你可以从后台日志中看到极其精彩的挣扎与自救过程：

1. 智能体首先尝试调用 get_flight_times。
2. 捕获到无情的 HTTP Error 404。
3. 智能体没有崩溃，也没有向用户抱怨。它依据系统指令，迅速“扭头”去调用了 `get_flight_times_backup`。
4. 备用接口成功返回数据，智能体面带微笑地向用户输出：“为您找到以下去巴厘岛的航班...”。

这就是生产环境中的优雅！通过配置备用工具和明确的容错指令，智能体挽救了一次本该失败的客户交互。

## 10.3 结语与前沿展望：大道至简的 SWE-agent

到这里，我们的课程内容已经全部结束。但作为本书的压轴，我必须带你跳出传统的框架，去看看目前开源界最前沿、也最能代表生产力极限的标杆项目——**SWE-agent**。

SWE-agent 是由普林斯顿大学和斯坦福大学的研究人员共同打造的 AI 软件工程师。它不是在聊天框里写两句代码的玩具，而是能够直接读取真实 GitHub 仓库的 Issue，自主配置环境、复现 Bug、修改代码并提交测试的终极“数字打工人”。

**为什么要提它？因为它的最新版本 `mini-swe-agent` 彻底颠覆了我们对“复杂智能体”的认知。**

在过去的一年里，大家都在疯狂地给智能体堆砌各种花哨的 API 接口和庞大的代码库。但 mini-swe-agent 返璞归真，做到了令人发指的极简：

- 百行代码：它的核心 Agent 类代码只有 100 行左右！
- 放弃花哨工具，独尊 Bash：它没有任何除了 Bash (命令行外壳) 之外的外部工具。它甚至懒得去使用大模型自带的 Tool-calling 接口。
- 一切皆子进程：它通过极其简单的 subprocess.run 独立执行每一次操作，这让它能极其丝滑地在各种沙盒环境（如 Docker）中运行并横向扩展。

**效果如何？** 就靠这 100 行代码和一个纯粹的 Bash 环境，搭配上顶尖的大模型（如 Claude 3.5 Sonnet 或 GPT-4o），它在权威的真实代码修复榜单 SWE-bench Verified 上，跑出了超过 **74%** 的惊人修复率！目前已有 Meta、NVIDIA、IBM 等众多科技巨头在内部使用它。

SWE-agent 的成功为全书画上了一个完美的句号，也留给了我们一个深刻的启示： 优秀的 AI 智能体设计，不在于你的架构图有多么庞大，也不在于你手写了多少个复杂的 API 工具。真正的王道在于：**深刻理解并释放大语言模型原生的推理能力，为它提供一个极简但极其自由的执行沙盒（如 Bash），剩下的，就放手让 AI 去大显身手吧！**

> **引用参考**
>
> *   **全链路评估 (Step-by-Step Evaluations)**: 生产环境中对智能体各个中间节点（如延迟、意图理解、工具选择准确度等）进行独立监控和测量的过程。
>
> *   **备用函数 (Backup Function/Fallback)**: 提高系统韧性的设计模式。当主要外部服务不可用时，智能体可自主切换至预先定义的备用服务。
>
> *   **SWE-agent / mini-swe-agent**: 普林斯顿与斯坦福大学开源的极简主义 AI 编程智能体。它摒弃了复杂的工具调用链，仅通过单一的 Bash 终端与环境交互，完美展示了“用最简单的沙盒跑最狂野的逻辑”的设计哲学。
